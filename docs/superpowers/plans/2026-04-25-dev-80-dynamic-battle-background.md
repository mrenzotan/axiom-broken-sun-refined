# DEV-80 — Dynamic Battle Scene Background Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Pair with the `executing-unity-game-dev-plans` skill for the Unity Editor handoffs and UVCS check-in cadence.

**Goal:** Make the Battle scene background swap dynamically based on where the player engaged combat in the platformer world, so each region shows its matching backdrop.

**Architecture:** A new `BattleEnvironmentData` ScriptableObject carries a background sprite + ambient tint. `BattleEntry` gains an optional `BattleEnvironmentData` reference. `ExplorationEnemyCombatTrigger` populates it from a serialized field on the trigger. `BattleController.Start()` reads it from `PendingBattle` and delegates to a pure-C# `BattleEnvironmentService` that applies it to the background `SpriteRenderer` before `Initialize()`. Fallback: if no environment data is provided, the existing static background remains unchanged.

**Tech Stack:** Unity 6.0.4 LTS, URP 2D, ScriptableObject data layer, Unity Test Framework (Edit Mode, NUnit).

**Related tickets:**
- **DEV-46** — Level 1 Snow Mountain (provides the Snow Mountain background sprite for the first authored asset)
- **DEV-81** — Per-encounter music changes (out of scope for v1; this ticket adds only the sprite infrastructure)

**Prerequisites:** None — this is greenfield infrastructure. No existing battle background system exists.

## Task 2: BattleEnvironmentService (plain C#)

**Files:**
- Create: `Assets/Scripts/Battle/BattleEnvironmentService.cs`
- Create: `Assets/Tests/Editor/Battle/BattleEnvironmentServiceTests.cs`

### Step 1: Write BattleEnvironmentService.cs

```csharp
using UnityEngine;
using Axiom.Data;

namespace Axiom.Battle
{
    /// <summary>
    /// Applies a BattleEnvironmentData SO to a SpriteRenderer.
    /// Pure C# — zero Unity lifecycle. Call from BattleController.Start() before battle begins.
    /// </summary>
    public sealed class BattleEnvironmentService
    {
        /// <summary>
        /// Applies the environment data to the given renderer.
        /// If either argument is null, silently no-ops (fallback to static background).
        /// </summary>
        public void Apply(BattleEnvironmentData environmentData, SpriteRenderer backgroundRenderer)
        {
            if (environmentData == null) return;
            if (backgroundRenderer == null) return;

            backgroundRenderer.sprite = environmentData.backgroundSprite;
            backgroundRenderer.color = environmentData.ambientTint;
        }
    }
}
```

                _enemyData     = pending.EnemyData;
                _battleEnemyId = pending.EnemyId;

## Architecture Notes

### Why `BattleEnvironmentService` is a plain C# class

Follows the project's non-negotiable rule: **MonoBehaviours handle Unity lifecycle only**. `BattleEnvironmentService` has no `Start`/`Update`/`OnDestroy` — it's a pure stateless function. `BattleController.Start()` (lifecycle) calls it.

### Why `SpriteRenderer` not Canvas `Image`

The Battle scene renders player/enemy sprites in world space with `SpriteRenderer` components (confirmed by `SpellVFXController` using `_spriteRenderer`). A world-space background `SpriteRenderer` on a low sorting layer is consistent with this approach. Using a Canvas `Image` would require resolving sorting order between Canvas and world-space sprites, which adds complexity.

### Why no `BattleEnvironmentCatalog`

Only one environment data instance is needed per region. A catalog (dictionary lookup) adds indirection without benefit at this scale. If regions exceed ~20 in Phase 6, a catalog can be added later — YAGNI for now.

### Why environment data is on the trigger, not the enemy

Different enemies in the same region share the same background. Putting the environment reference on the trigger (which is per-enemy-instance in the scene) means each placed enemy can be wired once in the Inspector. An alternative would be a per-scene default on `PlatformerWorldRestoreController`, but the trigger approach is simpler and already compatible with the existing arch.
